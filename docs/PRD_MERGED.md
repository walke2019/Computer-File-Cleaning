## 1. 产品愿景与目标

### 1.1 产品愿景

**"让每一个文件都有归属，让混乱的磁盘变得井井有条"**

打造一款 **隐私优先、全本地运行** 的 AI 文件智能整理工具，利用本地大语言模型（LLM）和多模态能力，实现文件的 **智能分析 → 自动分类 → 智能重命名 → 持续监控**。

### 1.2 问题陈述

在Windows生态中，用户长期面临磁盘空间不足、文件管理混乱、垃圾文件难以识别的问题。传统清理工具（如CCleaner、系统自带工具）往往基于固定的规则（后缀名、临时目录），缺乏对用户个人习惯的理解，容易产生"误删焦虑"，或者清理不够彻底，且缺乏智能分类整理能力。

### 1.3 核心价值主张

> **"不只是清理，更是智能整理"** - 100% 本地 AI 处理，保护隐私的同时释放存储空间

### 1.4 关键差异化优势

| 传统清理工具 | 我们的 AI 工具 |
|--------------|----------------|
| 规则式清理（删除所有 .tmp 文件） | 上下文理解（知道哪个 .tmp 是正在编辑的文档） |
| 仅字节级重复检测 | 语义相似性（50张相似日落照片分组） |
| 被动工具 | 主动助手（"你有6个月没打开这些文件了"） |
| 黑箱操作 | 透明解释（"为什么 AI 选择了这个文件？"） |

## 2. 市场定位与差异化

### 2.1 目标用户

| 用户群体 | 痛点 | 解决方案 |
|----------|------|----------|
| 普通用户 | 下载文件夹混乱、桌面堆满文件 | 一键智能整理、自动分类 |
| 专业人士 | 项目文件散乱、难以检索 | 智能建立项目结构、语义搜索 |
| 隐私敏感用户 | 不信任云端 AI 服务 | 100% 本地运行、离线可用 |
| 摄影/设计师 | 大量图片无序、命名混乱 | AI 看图分类、智能重命名 |

### 2.2 核心差异化

| 竞品不足 | 我们的方案 |
|----------|------------|
| 依赖云端 API，有隐私风险 | 全本地 LLM，零上传 |
| 仅支持文本文件 | 多模态：文本/图片/PDF/音频 |
| 规则式分类，不智能 | AI 语义理解，自适应学习 |
| 无法撤销，操作危险 | 完整的预览 + 撤销系统 |
| 界面复杂，需要技术背景 | 零配置开箱即用 |

### 2.3 竞品分析

| 竞品 | 模式 | 核心功能 | 优势 | 劣势 |

## 3. 用户画像

### 3.1 主要用户画像

#### 画像 1: 创意工作者 / 内容创作者 💰💰💰
- **特征**: 设计师、摄影师、视频剪辑师
- **痛点**: 渲染时"磁盘已满"错误；成千上万未整理的素材、重复项和项目备份
- **需求**: 可视化空间占用、找到遗忘的大文件、安全清理不破坏项目链接
- **付费意愿**: 高

#### 画像 2: 数字囤积者 💰💰
- **特征**: 普通办公人员，多年积累大量文件
- **痛点**: 被杂乱淹没；害怕删除重要内容导致根本不敢清理
- **需求**: 安心感、"安全"清理、分类整理（如"这些是3年前下载从未打开的表情包"）
- **付费意愿**: 中

#### 画像 3: 隐私倡导者 💰💰
- **特征**: 技术敏感用户，关注数据安全
- **痛点**: 追踪 Cookie、浏览器历史、文件元数据
- **需求**: 安全删除（粉碎）、元数据清理、离线处理
- **付费意愿**: 中

#### 画像 4: 休闲用户 / 游戏玩家 💰
- **特征**: 普通消费者，非技术背景
- **痛点**: 电脑运行慢、游戏卡顿
- **需求**: "一键修复"、设置后忘记
- **付费意愿**: 低

### 3.2 用户故事

| ID | 角色 | 用户故事 | 验收标准 |
|----|------|----------|----------|
| US-01 | 普通用户 | 作为非技术用户，我想要一个"一键清理"按钮，无需担心技术细节即可释放空间 | 1. 系统自动扫描安全删除区域<br>2. 用户看到简单的"立即清理"按钮<br>3. 释放空间且不删除个人文件 |

### 5.2 技术栈选型

| 层级 | 技术选择 | 理由 |
|------|----------|------|

| **核心逻辑** | TypeScript + Node.js | 类型安全、与 Electron 无缝集成 |
| **高性能模块** | Rust (via napi-rs) | 文件扫描、哈希计算等性能关键路径 |

| **数据库** | SQLite (better-sqlite3) | 轻量、无需安装、事务支持 |
| **打包** | electron-builder | 成熟的打包和自动更新方案 |

### 5.4 核心模块设计

#### 扫描引擎 (Scan Engine)
```typescript
interface ScanEngine {
  // 快速扫描（常用位置）
  quickScan(): Promise<ScanResult>;
  
  // 深度扫描（全盘）
  deepScan(paths: string[]): Promise<ScanResult>;
  
  // 增量扫描（仅变化文件）
  incrementalScan(): Promise<ScanResult>;
  
  // 实时监控
  watch(paths: string[]): FileWatcher;
}
```

#### AI 分析引擎 (AI Analyzer)
```typescript
interface AIAnalyzer {
  // 文件分类
  categorize(file: FileInfo): Promise<Category>;
  
  // 重要性评分 (0-10, 10=安全删除)
  scoreRisk(file: FileInfo): Promise<number>;
  
  // 图像相似度分组
  groupSimilarImages(images: FileInfo[]): Promise<ImageGroup[]>;
  
  // 自然语言查询
  query(prompt: string): Promise<FileInfo[]>;
}
```

#### 清理引擎 (Clean Engine)
```typescript
interface CleanEngine {
  // 移动到隔离区（可恢复）
  quarantine(files: FileInfo[]): Promise<void>;
  
  // 永久删除
  delete(files: FileInfo[]): Promise<void>;
  
  // 安全粉碎（多次覆写）
  shred(files: FileInfo[]): Promise<void>;
  
  // 撤销上次操作
  undo(): Promise<void>;
}
```

### 6.2 核心交互流程

```
┌──────────────┐     ┌───────────────┐     ┌───────────────┐
│  选择目录     │ ──▶ │  AI 分析中...  │ ──▶ │  预览结果页    │
│  拖放/点击    │     │  进度条+预览   │     │  分类/重命名   │
└──────────────┘     └───────────────┘     └───────────────┘
                                                   │
                                                   ▼
┌──────────────┐     ┌───────────────┐     ┌───────────────┐
│  操作完成     │ ◀── │  确认执行      │ ◀── │  用户调整      │
│  可撤销       │     │  一键应用      │     │  修改/排除     │
└──────────────┘     └───────────────┘     └───────────────┘
```

### 6.3 设计原则

- **信任优先**: 每个破坏性操作都需要明确的确认或可见的"撤销"路径
- **透明度**: 没有"黑箱"魔法。解释*为什么*标记要删除某个文件
- **现代原生**: 使用 Fluent Design (Windows) / Apple Human Interface Guidelines (macOS) 美学

### 6.4 无障碍要求

- **AR-01**: 所有交互元素支持完整的键盘导航（Tab、Enter、Esc、方向键）
- **AR-02**: 屏幕阅读器兼容性（所有按钮上的 ARIA 标签，尤其是仅图标按钮）
- **AR-03**: 高对比度模式支持，符合 WCAG 2.1 AA 标准

## 10. 隐私承诺

### 10.1 核心原则

1. **零上传**: 所有 AI 处理 100% 本地完成
2. **无遥测**: 不收集任何用户数据
3. **开源透明**: 核心代码开源可审计
4. **离线可用**: 无需网络连接即可工作
5. **数据自主**: 所有配置/日志存储在本地

### 10.2 安全特性

- **本地化运行**: 所有AI分析（如图片识别、文本分析）必须**完全在本地运行**，严禁上传用户文件内容到云端
- **沙箱机制**: 清理操作在模拟环境中预演，确保不破坏系统依赖
- **代码开源（核心逻辑）**: 提供详细的扫描日志，界面显著位置标注"本地运行，不联网上传"

## 11. 风险评估

### 11.1 风险矩阵

| 风险 | 可能性 | 影响 | 缓解策略 |
|------|--------|------|----------|
| **技术风险**: 本地 AI 性能不足 | 中 | 高 | 分层模型策略（快速模型 + 精确模型） |
| **技术风险**: Electron 内存占用高 | 高 | 中 | 考虑迁移到 Tauri (Rust) |
| **市场风险**: Windows 内置功能增强 | 中 | 高 | 差异化 AI 功能、专注高级用户 |
| **竞争风险**: 巨头进入市场 | 低 | 高 | 快速迭代、社区建设 |
| **运营风险**: 误删用户重要文件 | 低 | 极高 | 强制隔离期、多重确认、完善撤销 |

### 11.2 关键缓解措施

1. **数据安全第一**: 任何删除操作默认进入 7 天隔离期
2. **透明 AI**: 所有 AI 决策提供解释，用户可手动覆盖
3. **渐进权限**: 首次仅请求用户文件夹权限，按需请求更多
4. **引入"隔离区"机制**: 删除文件保留7-30天
5. **建立白名单数据库**: 系统文件、常用软件配置
6. **默认策略为保守型**: False Negative > False Positive

### 11.3 技术风险缓解

| 风险 | 缓解措施 |
|------|----------|
| 本地模型性能不足 | 提供多档位模型选择 |
| AI 分类错误 | 必须预览确认，可撤销 |
| 跨平台兼容性 | Electron 统一 + 平台适配层 |
| 模型安装复杂 | 内置轻量模型 / 一键安装 Ollama |
| Python/AI 模型在Windows客户端打包体积过大，启动慢 | 使用轻量化模型 (如 MobileNet, TinyBERT)，考虑 ONNX Runtime 加速，核心IO逻辑使用 Rust/C++ 编写，UI使用 Electron 或 Flutter |

### 11.4 信任危机缓解

- **风险**: 用户不相信AI的判断，认为在窃取隐私
- **缓解**: 代码开源（核心逻辑），提供详细的扫描日志，界面显著位置标注"本地运行，不联网上传"
